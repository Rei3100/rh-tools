# 自動並び替え 精度オーバーホール 設計書

作成日: 2026-06-23

## このラウンドの目的

自動並び替えを「**信頼して任せられる**」精度に引き上げる。
現状は依存関係（dependencies）だけを揃えるトポロジカルソートで、「同じファイルを触るMODのどちらが反映されるか（＝競合）」は人任せ。結果「有効にしたのに反映されてない」が起きる。

ゴール：**MODを入れて有効にするだけで、ソフトが各MODの"ゲームの何を触るか"を読み取り、競合を解いて最適な順序に自動配置する。** ユーザーは何も判断しなくてよい。

横断ルールは [信頼性オーバーホール設計](2026-06-22-reliability-overhaul-design.md) を継承：
1. 実行時に Claude を呼ばない（知識はソフト内のルール＋解析に持つ。拡張型）
2. 失敗・変更を握りつぶさない（必ずログ＋画面に残す）
3. 自動修正前に必ずバックアップ、ワンタップで元に戻せる
4. 本体側では何もしない・全部ヘルパーに集約

## 確定した方針（ブレストでの合意）

- **理想は全自動。** ユーザーの本音は「何も判断せず、全部自動で最適配置」。手間ゼロ・余計な思考ゼロが最優先。
- **ただしハイブリッド。** ソフトが本当に判断できない競合だけ一度聞き、その選択を記憶 → 二度と聞かない。ユーザーの意図が資産化される。
- **対象は"全部"。** 物理的・仕組み的に不可能なもの以外は、可能な限り賢く読む。今回は4種のアナライザを一度に組む。
- **発動タイミング：起動時・有効/無効切替時・削除時、必要な場面で必ず。** ユーザーが意識しなくても常に最適順を保つ。
- **見せ方：診断パネルに履歴。** 普段は黙る。何をしたかは診断画面の「自動配置履歴」で後から確認できる。

## アーキテクチャ

### 1. アナライザ基盤（`IModAnalyzer` 拡張ポイント）

各MODを解析し「このMODが触る**資源**のリスト（事実）」へ正規化する。資源は型を持つキーで表す：
- ファイル資源：`file:<相対パス>`（既存の `ModContentScanner` の出力形式を踏襲）
- 曲資源：`song:<曲ID>`
- コスチューム資源：`costume:<キャラ/枠ID>`

すべて「資源キー → それを触るMOD」に正規化することで、競合検出・並び替えは資源の種類が増えても同一ロジックで動く。

| アナライザ | 読む対象 | 出す資源 | 状態 |
|---|---|---|---|
| `FileReplaceAnalyzer` | CPK/AWB 等の redirect ルート配下のファイルツリー | `file:*` | 既存 `ModContentScanner` を IModAnalyzer 化・ルート拡張 |
| `BgmeAnalyzer` | BGME 設定（曲ID定義ファイル） | `song:*` | 新規 |
| `CostumeAnalyzer` | コスチュームフレームワークの登録ファイル | `costume:*` | 新規 |
| `StructureAnalyzer` | フォルダ構造 vs 期待形 | 構造警告（資源ではなく診断フラグ） | 新規 |

`StructureAnalyzer` だけは競合ではなく「置き場所が変」の指摘を出す（後述の診断側へ）。

> 各アナライザの「読む対象」フォーマット（BGME の曲ID定義の実ファイル形式、コスチューム登録の実形式）は、実機MODのサンプルを採取して TDD で確定する。形式が読めないMODは「未対応」として握りつぶさず記録する（誤った事実を作らない）。

### 2. 役割分類（`ModRoleClassifier`）

競合の勝者を決める知識。セクションAで取得済みの **GameBananaカテゴリ＋ファイルパスのパターン** から各MODの役割を推定：

| 役割 | 例 | 競合時の扱い |
|---|---|---|
| 見た目上書き（VisualOverride） | 髪型・テクスチャ・コスチューム見た目 | **勝たせたい**（後勝ち側へ） |
| 土台（BaseLayer） | 体型・大型オーバーホール・下地 | 負けてよい（前へ） |
| 音楽（Music） | BGME 系 | 別枠ルール（曲ID単位で判定） |
| ライブラリ/フレームワーク（Library） | 依存される基盤 | 依存ルールで前方固定 |
| 不明（Unknown） | 分類できない | 自動判断せず「要確認」候補 |

この役割テーブルがソフトの知識資産。後からルール（カテゴリ→役割のマッピング、パスパターン）を**足すだけで賢くなる**。

### 3. 決定エンジン（`LoadOrderOptimizer`／既存 `LoadOrderSorter` を拡張）

最終的な読み込み順を決める。優先度（上が強い）：

1. **依存関係**（既存トポロジカルソート）— 必須前提。絶対に守る。
2. **学習した好み**（`PreferenceStore`）— ユーザーが過去に決めた競合ペアの勝敗。ルールより優先。
3. **役割ルール**で競合の勝者を自動決定 → 勝者を後ろ（後勝ち）に置く。
4. 上記で順序が定まらない部分は、現在順を安定的に保持（既存の stable ordering）。

出力は「並び替え後の順序」＋「何を・なぜ動かしたか（配置理由）」のリスト。後者が診断パネルの履歴になる。

> 後勝ち（読み込み順で最後＝優先）の前提は D+E ラウンドで実機サニティチェック済みの想定。逆だった場合は勝者判定の1箇所を反転するだけで済む構造にする。

### 4. 好みの記憶（`PreferenceStore`）

- ソフトが本当に判断できない競合（役割が Unknown 同士、または同役割で甲乙つけがたい）だけ、ユーザーに一度だけ問う。
- 「MOD-A を MOD-B より優先」という**競合ペアの勝敗**を `%APPDATA%\ReloadedHelper` に保存。
- 同じペアの競合は二度と聞かず、保存値を使う。
- **手動ドラッグも好みシグナルとして扱う**：ユーザーが手で順序を入れ替えたら、その勝敗を PreferenceStore に記録し、以後の自動並び替えがそれを尊重する（自動が手動と喧嘩しない）。

### 5. 自動配置の発動（`AutoSortCoordinator`）

発動タイミング（合意済み）：
- **アプリ起動時** … 常に最適順から始める（自己修復）。
- **有効/無効の切替時** … そのMODを最適位置へ即挿入。
- **MOD削除時** … 残りを最適順へ。
- 「全件強制取り直し」等で前提が変わったとき。

各発動で：
1. 並び替え前に既存の `LoadOrderBackupService` でバックアップ。
2. `LoadOrderOptimizer` で新順序＋配置理由を算出。
3. 適用。
4. 配置理由を**診断パネルの「自動配置履歴」**に追記（普段はトースト等で止めない）。
5. ワンタップで直前のバックアップへ戻せる導線を診断パネルに置く。

判断できない競合が出た場合だけ、その時に一度問う（手間ゼロの原則を破らない範囲で最小限）。

### 6. 「なぜ効かない」診断の精度向上（既存 `ModDiagnostics`／`GameDiagnostics` 拡張）

4アナライザ＋役割情報を入力に、原因判定を増強：
- 依存フレームワークが無効（既存）
- 対象ゲーム違い（AppId 不一致）（既存）
- **上書き負け：犯人MOD名を特定**して提示（既存の競合集約を、役割で「これは負けてよい/おかしい」まで判定）
- **置き場所が変**（`StructureAnalyzer` の構造警告）
- **曲ID被り**（`BgmeAnalyzer` の `song:*` 競合）

## 新規/変更クラス（想定）

- `IModAnalyzer`（新規・拡張ポイント）… 解析の共通インターフェース。
- `FileReplaceAnalyzer`（既存 `ModContentScanner` を移行）… `file:*` 抽出。redirect ルート拡張。
- `BgmeAnalyzer`（新規）… `song:*` 抽出。
- `CostumeAnalyzer`（新規）… `costume:*` 抽出。
- `StructureAnalyzer`（新規）… 構造警告。
- `ModRoleClassifier`（新規）… カテゴリ＋パス → 役割。
- `LoadOrderOptimizer`（新規 or `LoadOrderSorter` 拡張）… 依存＋好み＋役割で順序＋配置理由を算出。
- `PreferenceStore`（新規）… 競合ペアの勝敗を永続化。
- `AutoSortCoordinator`（新規 or App 内）… 発動タイミング制御・バックアップ・履歴記録。
- `ConflictDetector`（既存）… 資源キー汎用化（file 以外も扱う）。
- `ModDiagnostics`/`GameDiagnostics`（既存拡張）… 新原因・履歴の提示。
- App 側：診断パネルに「自動配置履歴」＋「元に戻す」導線。

## エラーハンドリング方針

| 状況 | 対処 |
|---|---|
| アナライザが形式を読めない | 「未対応」として記録（握りつぶさない）。事実を捏造しない |
| 役割が判定できない競合 | 自動で決めず、一度だけユーザーに問う → PreferenceStore へ |
| 並び替え適用失敗 | バックアップから復帰、診断パネルに失敗理由を記録 |
| 循環依存 | 既存どおり末尾に原順で残す |

## テスト方針

- Core 層（各 Analyzer・`ModRoleClassifier`・`LoadOrderOptimizer`・`PreferenceStore`・`ConflictDetector`）は xUnit。
- BGME/コスチュームの解析は、実機MODから採取した固定サンプルに対して検証（TDD）。
- 並び替えは「依存 > 好み > 役割」の優先度が守られることをケースで固める。
- UI（診断パネル履歴・問い合わせダイアログ）は目視確認。

## スコープ外（将来反復）

- クラッシュ検知（Windows イベントログ）… 別ラウンド。
- アナライザ未対応の特殊MOD仕組み（順次追加）。
- 役割ルールの本格的な学習自動化（今回は手動ルール＋ペア記憶まで）。
