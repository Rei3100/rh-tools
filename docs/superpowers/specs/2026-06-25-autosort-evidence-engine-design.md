# 自動配置 信頼性リフレーム（証拠ベース競合エンジン）設計書

作成日: 2026-06-25

## このラウンドの目的

自動並び替え・自動配置を「**完全に信頼して任せられる**」精度へ作り替える。
3セッション、種類分類（ModType）を磨き続けて頭打ちになった。原因は**設計の形**：
「MODを種類に仕分けし、種類の固定順（土台→見た目）で全体を並べる」やり方が、
**本当に効く"具体的な競合"ではなく、その代用品（種類ラダー）で順序を支配している**こと。

このラウンドで、順序を決める主役を **「種類の順番」から「実際の依存と競合＋複数の証拠」へ移す。**

## 北極星（揺らさない原則）

- **完全自動。** ユーザーが触るのは **導入・有効/無効・設定変更だけ**。
- **並び替えはユーザーがやらない・できない。** 手動並び替えUIを持たない。エンジンが**必ず最後まで順序を決め切る**。
- **「どっちを勝たせる？」とユーザーに聞かない。** 現状の "要確認（Unresolved）" のような順序の丸投げは**廃止**。
- **ユーザーに見せる競合はただ1種類** ＝「**この2つはどちらか片方向け（変種/冗長）。両方有効でも片方は無意味**」。解決は**無効化のみ**、並び替えはさせない。
- **表示ソートと実配置は独立。** 一覧はカテゴリ/作者/名前で**閲覧用に並べ替え可**だが、実際の読み込み順は不変（エンジンが管理）。
- **安心の根拠を見せる。** 介入しない代わりに「ゲーム内でこう反映される」を日本語で予告する。

### 継承する不可侵ルール（信頼性オーバーホールから）

1. 実行時に Claude / 外部AI を呼ばない（知識はソフト内のルール＋解析。無料・オフライン）。
2. 失敗・変更を握りつぶさない（必ずログ＋画面に残す）。
3. 自動修正の前に必ずバックアップ、ワンタップで元に戻せる。
4. ゲーム本体側では何もしない・全部ヘルパーに集約。
5. ユーザーデータは `%APPDATA%\ReloadedHelper` のみ。

## 現状の限界（なぜ作り替えるか）

現在の `LoadOrderOptimizer.Optimize` の流れ：

1. **種類ランクで全MODを大域整列**（`ModTypeInfo.Rank`）← ここが諸悪の根源
2. 依存トポロジカルソート（`LoadOrderSorter`）
3. 競合ごとに勝者を決定（2件は好み、その他は種類で一意なら）→ 動かす
4. 決められない競合は "要確認" としてユーザーへ

| 限界 | 症状（ユーザーの言葉） |
|---|---|
| 種類ラダーが**大域順序を支配** | 「②動かしすぎ・予想外」「触ってないのに大量に並び替わる」 |
| 勝者の根拠が**種類だけ**で弱い | 「①並び順が時々おかしい」 |
| 決められない競合を**ユーザーに丸投げ** | 「③要確認が多い」＝全自動の理想に反する |
| **依存グラフ・"効いてる証拠"を未活用** | 「④合ってるか分からない」 |

## 新アーキテクチャ（A–E）

### A. 依存を背骨に、種類は「実際に触る資源」から

**資源（実際にゲームの何を書き換えるか）を最優先の真実とする。** 既存アナライザ
（`FileReplaceAnalyzer` / `BgmeAnalyzer` / `CostumeAnalyzer`）が出す `ResourceKey`
（`file:` / `song:` / `costume:`）が、そのMODの「触る部品リスト」。これが競合検出にも種類推定にも使える。

- **種類推定の優先順位（修正後）**
  1. **実際に触る資源**（`song:` を書くなら音楽性、`costume:` ならコスチューム性、モデルファイルならモデル）← 主軸
  2. **依存グラフ**（フレームワーク vs コンテンツの区別、読み込み順の制約。**ただし種類の断定には使わない**——大型MODは音楽/衣装/モデル枠組みに同時依存するため）
  3. **カテゴリ/キーワード**（`ModTypeClassifier` の既存ロジック）← 最後の保険
- **フレームワーク/ライブラリの確定**：`IsLibrary`、または「多数のMODから依存される」もの → 前方固定（既存トポロジカルソートで担保）。依存マップ（`ModConfig.json` の `ModDependencies` ＋ `PluginData.GameBananaDependencies/GitHubDependencies`）から機械的に判定。
- 役割マッピング表（既知フレームワーク → ヒント）は**データ資産**として持ち、足すだけで賢くなる。

### B. 競合駆動の最小移動（大域整列の廃止）

**`Optimize` の「種類ランクで全体を並べる」ステップ0を撤去する。**

1. 起点 = **現在の順序**。
2. **依存を満たす**（`LoadOrderSorter`、既存のまま）。
3. そのあと、**実際に同じ資源を奪い合うMODだけ**を動かして競合を解消する。
4. **競合に関与しないMODは一切動かさない** → churn（無駄な並び替え）ゼロ・挙動が安定・説明可能。

> **大域的な種類整列は行わない。** 種類は C の競合解決の中でのみ「弱い tie-break」として使う。資源を共有しないMODの相対順は**現在順のまま**触らない。新規に有効化したMODは末尾付近に置き、競合する相手に対してのみ移動する。人間が見やすい並びは「表示ソート」（実配置とは独立）で担保する。

### C. 証拠で勝者を決める（必ず決め切る）

各競合（同じ資源を触るMOD群）について、**勝者＝後ろ＝反映される側**を以下のスコアで決める（強い順）。

1. **依存の向き**：A が B に依存（B が土台）なら、消費側 A を後ろ（上書き側）に。ただし依存制約を壊さない範囲。
2. **アドオン/変種関係**：名前・宣言が `addon`/`companion`/`patch`/`変種`、または相手MODに依存 → 後ろ（拡張する側を勝たせる）。
3. **ピンポイント度（specificity）**：触る資源が**少ない**ほど「狙った上書き」とみなし後ろへ（例：双葉の顔1枚だけ差し替えるMOD ＞ 双葉まるごと大型MOD）。
4. **作者の配置指示**：説明文・同梱txt の `load below` / `at the bottom` / `下に置く` / `priority` 等を**限定パターン**で抽出（強いが絶対ではない証拠）。
5. **種類ランク**：`ModTypeInfo.Rank` を**弱い tie-break に降格**。
6. **過去のユーザー操作**：あれば尊重（後述 `PreferenceStore` の転用）。

- 上記で決着しない同点でも、**安定規則（現在順を保持等）で必ず一意に決める**。"要確認" は出さない。
- すべての決定は**根拠つきで履歴に残す**（監査可能）。根拠が弱い決定は「弱い根拠」と正直にフラグする（握りつぶさない）。

### D. "片方だけ" 検出（唯一のユーザー向け競合）

「両方有効にしても片方が完全に無意味」になるペアだけを、**非ブロッキングに**提示する。

- **検出条件（いずれか）**
  - 整列後、**片方の資源がほぼ全て相手の影に隠れる**（実質ゼロ寄与）。
  - **兄弟変種シグナル**：同一の土台MODに依存＋命名が変種関係（例：`kasumi.altmodels` と `kasumi.altmodels.longhairvariant`）＋資源が高度に重複。
- **提示**：「この2つは"どちらか片方"向けです。両方有効でも片方は無意味になります。片方を無効にしますか？」（モーダルで作業を止めない／一覧上のさりげない指摘として）
- **解決は無効化のみ。** 並び替えはさせない。
- **却下を記憶**：「両方残す」を選んだら二度警告しない。

### E. "効いてる証拠" レイヤー（ログ不要）

エンジンは全資源と全勝敗を知っているので、**結果を日本語で予告**できる。

- **可視結果の予告**：例「ゲーム内では『黒髪双葉』が反映されます（双葉の顔テクスチャ2枚を『カナミ双葉』と取り合い、後勝ち）」。ユーザーはゲームで現物を確認できる＝**ログ無しで安心の根拠**。
- **診断パネルの強化**：変更内容／各競合の勝者と根拠／"片方だけ"／分類不能（未対応）件数／"根拠が弱い"件数を正直に表示。普段はここで静かに確認できる。
- **起動時ログ/クラッシュ検知は将来枠**：Reloaded-II は**ローダーログをファイル保存しない**ことを実機確認済み（起動時コンソールへ流すのみ）。確実な常時取得が難しいため本ラウンドの対象外。代わりに本レイヤーで先に"安心"を作る。

## 表示と実配置の分離

- 一覧は**カテゴリ/作者/名前で表示ソート可**（閲覧用）。これは見た目だけで、**実際の読み込み順は変えない**。
- 実配置（読み込み順での位置）と配置理由は**別表示**で確認できる。
- **手動並び替え／ドラッグ並べ替えUIは持たない**（既存の未配線 `LearnFromManualOrder` 経路は不要として撤去 or 封印）。

## 変更・新規・再利用クラス

**再利用（ほぼそのまま）**
- `Analyzers/*`（`IModAnalyzer`・`ResourceKey`・`FileReplaceAnalyzer`・`BgmeAnalyzer`・`CostumeAnalyzer`・`StructureAnalyzer`）— 資源抽出基盤。
- `ConflictDetector` — 資源キー競合検出。
- `LoadOrderSorter` — 安定トポロジカルソート。
- `AutoSortCoordinator` — バックアップ＋適用＋履歴。
- `GameDiagnostics` — 解析パイプラインの組み立て点。

**変更**
- `LoadOrderOptimizer` — **大域整列ステップ0を撤去**。競合駆動の最小移動＋証拠スコア（C）に置換。`Unresolved` の概念を廃止（必ず決め切る）。
- `ModTypeClassifier` — **資源主軸**に。依存は背骨/制約として使い、種類断定には使わない。keyword は保険。
- `PreferenceStore` — 「勝者の問い合わせ記憶」から **「却下した"片方だけ"警告の記憶」** へ転用（順序の好みは原則エンジンが決めるため）。
- `ModDiagnostics` — 競合メッセージの**「順序を入れ替えてください」文言を撤去**（ユーザーは並べ替えない）。"片方だけ" と "効いてる証拠" の文面へ。
- `MainViewModel.RunAutoSort` — 新エンジンへの配線。表示ソートと実配置の分離。

**新規**
- 役割マッピング表（既知フレームワーク → 種類ヒント、データ）。
- `RedundancyDetector`（D：冗長/変種ペア検出）。
- `EffectPredictor`（E：可視結果の予告文生成）。
- 勝者スコアリング（C：`WinnerEvidence` 等の内部構造）。

## 勝者方向（last-wins）の前提

- 既存どおり「**読み込み順で後＝勝ち**」を採用（`ConflictDetector` は `mods[^1]` を勝者）。
- **実機で各枠組み（CriFs/AWB/PAK/BGME/Costume）について1回ずつサニティ検証**する（実装の最初のタスク）。逆だった枠組みがあれば、勝者判定を**1箇所**に集約してあるので反転で対応。

## エラーハンドリング（握りつぶさない）

| 状況 | 対処 |
|---|---|
| アナライザが形式を読めない | 「未対応」として記録。事実を捏造しない（既存 `ModAnalysis.Failures`） |
| 勝者の根拠が弱い | それでも決め切るが「弱い根拠」とフラグして履歴/診断に表示 |
| 並び替え適用失敗 | バックアップから復帰、診断パネルに失敗理由 |
| 循環依存 | 末尾に原順で残す（既存） |

## テスト方針

- Core 層（新 `LoadOrderOptimizer`・種類推定・`RedundancyDetector`・`EffectPredictor`・勝者スコア）は xUnit。
- **実機サンプルで固定ケース**：かすみ変種（`altmodels` / `longhairvariant`）、双葉の髪（`Black Hair Futaba` / `CBT_Black_Hair_Futaba` / `futabahair` / `KanamiFutaba`）、BGME 音楽、コスチューム。
- **回帰で固定**：「競合に関与しないMODは動かない（churn ゼロ）」。
- 証拠スコアの**優先順位**がケースで守られること。
- UI（診断パネル・"片方だけ"提示・予告文）は実機目視。

## 実装の段階（成果を早く出す）

- **フェーズ1：順序の"正しさと安定"**（A・B・C）
  - B（大域整列撤去＝churn停止）→ A（依存背骨＋資源主軸の種類）→ C（証拠で勝者）。
  - これで「①順序ミス」「②動かしすぎ」「③要確認」が同時に改善。
- **フェーズ2：安心の可視化**（D・E）
  - D（"片方だけ"検出）→ E（予告＋診断強化）。「④合ってるか分からない」を解消。
- 各フェーズ末に**実機検証**（`-p:Version=9.9.9` で自動更新抑止のローカルビルド）。
