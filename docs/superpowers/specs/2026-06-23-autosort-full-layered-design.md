# 自動並び替え：層別フル整列エンジン 設計仕様

- 日付: 2026-06-23
- 対象: ReloadedHelper.Core（ロジック層）＋ App（表示のみ最小限）
- 状態: 承認済み（実装計画へ）

## 1. 背景と問題

自動並び替えの「精度向上」を繰り返しても、100以上あるMODのうち起動時に並び替えられるのは数件（実測3件）に留まり、実用にならない。

### 根本原因（コードで確認済み）

現行の `LoadOrderOptimizer.Optimize` は「並び替え」ではなく「衝突の仲裁」しかしない。MODを動かすには次の4関門を**すべて**通過する必要がある：

1. **同じファイルの奪い合いを検出できる** — だが検出元 `ModContentScanner` は `P5REssentials/CPK`・`FEmulator/AWB` の2ルート、加えて `BGME/*.pme`・`Costumes/` の計4種類の構造しか見ていない。それ以外の構成のMODは中身を一切見られず、衝突が永遠に検出されない。
2. **衝突が実際に重なっている**（同じ曲ID・衣装スロット・ファイル）。
3. **役割で勝者が一意に決まる** — 役割は `ModRoleClassifier` が GameBanana カテゴリ名のホワイトリストから決める。リスト外・カテゴリ無しは全部 `Unknown` → 「要確認」→ 動かさない。
4. **勝者がまだ負け側より前にいる**。

つまり「3件」は「悪い順を直せなかった」のではなく、**97件はそもそも評価対象に入っていない**ことを意味する。精度を上げても土俵に乗るMODが増えなければ変化は出ない。

### メタデータが当てにならない理由（コードで確認済み）

役割の素となる Category は GameBanana 照合できたMODにしか付かない。さらに：

- `GameBananaClient.ExtractIdFromUrl` は `gamebanana.com/mods/` か `/dl/` しか拾わない。`sounds/`・`wips/` 形式のURLは無視。
- `FetchAsync` は常に `/Mod/{id}` を叩く。Sounds・Wips は別種別なのでIDが取れても情報が来ない。
- GitHub 由来のMODは GameBanana ですらなく、一生取得されない。

→ カテゴリ情報は構造的に欠落するMOD群が存在する。**役割判定はMODの中身（フォルダ構成）を一次情報にしなければならない。**

## 2. ゴール

- 起動時に**全MODが意味のある通し順に整列**される（一部の衝突仲裁だけでなく全体整列）。
- カテゴリ情報が無いMODにも役割が付く（内容ベース分類）。
- 各MODに「なぜこの位置か」が表示され、納得感・信頼が得られる。
- 既存のバックアップ／復元で安全性を担保する。

### 非ゴール（今回はやらない）

- sounds/wips/GitHub のオンラインメタデータ取得対応（内容ベース分類があれば並び替えには不要。欲しければ別タスク）。
- 適用前プレビュー承認UI（将来の拡張候補。今回は全件理由表示で透明性を担保し、自動適用を維持）。
- Persona 以外のゲーム向けの汎用化（ユーザーの対象は Persona 系のため当面 Persona 中心で良い）。

## 3. 並びの理屈（層モデル）

Reloaded-II は「リストの後ろほど強い（上書きで勝つ）」。よって弱い土台を前、勝たせたい見た目を後ろに置く。層ランク（小さいほど前＝弱い）：

| ランク | 層 | 役割(ModRole) | 例 |
|---|---|---|---|
| 0 | ライブラリ／前提 | Library | フレームワーク、依存される側 |
| 1 | 大型改修・システム・キャラ追加 | BaseLayer | ゲームプレイ改変、大型オーバーホール |
| 2 | 音楽・サウンド | Music | BGME 等 |
| 3 | 見た目の上書き | VisualOverride | スキン／テクスチャ／UI／衣装／モデル |
| 4 | 個別パッチ（不明含む末尾寄せ） | Unknown(扱い) | 細かい上書き、判定不能 |

> 注: `Unknown` の最終的な配置ランクは実装時に確定する（暫定はランク4＝末尾寄せ）。衝突仲裁では従来どおり `Unknown` を自動判断材料にはしない。

## 4. アーキテクチャ

すべて Core 層。各ユニットは単一責務・独立テスト可能とする。

### 4.1 ContentRoleClassifier（新規）

責務：MODの中身と既存シグナルから `ModRole` と理由文字列を返す。

```
RoleDecision Classify(ModInfo mod, string? category)
  // RoleDecision = (ModRole Role, string Reason)
```

優先順位：

1. `mod.IsLibrary` が真 → `Library`（理由「ライブラリ指定のため」）。
2. フォルダ構成シグナル（`mod.FolderPath` を走査）：
   - `BGME/` が存在 → `Music`。
   - `Costumes/` のみが置換実体 → `VisualOverride`。
   - 既知のテクスチャ／スキン系置換ルート配下にファイル → `VisualOverride`。
   - 大量のファイル置換・改修マーカー → `BaseLayer`。
   （判定ルールの具体的な閾値・ルート一覧は実装計画で確定）
3. 2で決まらず GameBanana カテゴリがある → 既存 `ModRoleClassifier.Classify` を内部利用して補助判定。
4. それでも不明 → `Unknown`（理由「役割を判定できないため末尾寄せ」）。

既存 `ModRoleClassifier`（カテゴリ→役割表）は破棄せず、本分類器の一段として再利用する。

### 4.2 層別フル整列（LoadOrderOptimizer 拡張 or LayeredSorter 新規）

責務：依存を満たしつつ、全MODを層ランク順に安定整列し、各MODへ理由を付ける。

手順：

1. 各MODの層ランクを役割から決定。
2. 全MODを `(層ランク 昇順, 元の相対順 保持)` で安定ソート（同層は元の並びを維持＝無用な攪拌をしない）。
3. 依存制約を適用：前提MOD（dep）は依存元（dependent）より前。違反する場合は依存を優先して補正。
4. 既存の衝突解決（`prefs` の好み → 役割による勝者決定）を上に重ねて微調整。`Unknown` は従来どおり自動判断しない＝「要確認」。
5. 出力を拡張：従来の `Reasons`（動いたペア）に加え、**全MODの配置理由**（層＋根拠）を返す。

`OptimizeResult` を拡張、または新レコードで全件理由（`PlacementExplanation` 等）を持たせる。`Unresolved`（要確認）は維持。

### 4.3 スキャナのカバレッジ拡張（ModContentScanner）

責務：実際の競合をより多く検出できるよう、認識する置換ルートを拡張する。Persona 中心で良い。具体的な追加ルート一覧は実装計画で確定。

### 4.4 UI：全件「なぜここ」表示（App、最小限）

MOD一覧の各行に配置理由を表示（ツールチップまたは小さなラベル）。Core が返す全件理由をそのまま使う。

## 5. データフロー

```
起動 / トグル / 削除 / 強制更新
  → MainViewModel.RunAutoSort
    → GameDiagnostics.Run（衝突検出: 拡張スキャナ＋アナライザ）
    → BuildRoles（ContentRoleClassifier で全MODに役割＋理由）
    → LoadOrderOptimizer.Optimize（層別フル整列＋衝突解決＋全件理由）
    → 順序が変わったら：Backup → AppConfig 書込 → 履歴保存 → RebuildEntries
  → UI が全件理由を表示
```

## 6. エラー処理

- フォルダ読込失敗・解析失敗：従来どおり空を返して握りつぶさない（アナライザは失敗を記録）。
- 破損JSON（履歴・設定）：既存のガードを維持。
- 書込前に必ず `LoadOrderBackupService.Backup`。
- 順序が変わらない場合は書込・履歴を作らない（既存挙動を維持）。

## 7. テスト方針（TDD）

- `ContentRoleClassifier`：各シグナル（Library / BGME / Costumes / テクスチャ / 大型 / カテゴリ補助 / 不明）ごとに role と理由を検証。
- 層別整列：層順の並び、同層での安定性、依存制約の維持、衝突解決との両立、全件理由の出力。
- 拡張スキャナ：新ルートの検出、既存ルートの非回帰。
- 統合：`MainViewModel.RunAutoSort` で「100件相当のうち多数が動く」ことを代表データで確認。
- 既存テストの非回帰（`LoadOrderOptimizerTests` ほか）。

## 8. リスクと対策

| リスク | 対策 |
|---|---|
| 全件大移動でユーザーが驚く／意図した手動順が壊れる | 全件理由表示で透明化＋バックアップ復元。同層は元順保持で攪拌最小化。 |
| 内容ベース分類の誤判定 | 理由を必ず表示し、誤りを目視可能に。`Unknown` は末尾寄せに留め断定しない。 |
| Persona 前提のルール拡張が他ゲームで的外れ | 非ゴールとして明示。Persona 中心で進める。 |
