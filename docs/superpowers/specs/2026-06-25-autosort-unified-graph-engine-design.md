# 自動配置エンジンv2「統一制約グラフ」設計書

作成日: 2026-06-25

## このラウンドの目的

自動配置エンジンを、世界標準の自動ロード順アルゴリズム（**LOOT** = Load Order Optimisation Tool、Vortex が内部採用）に学んだ**統一制約グラフ**へ作り替える。
現状のエンジン（フェーズ1）は「MODを種類のバケツに仕分け → 競合だけ後から動かす」という ad-hoc な作りで、実機検証で分類バグ（枠組みを音楽/戦闘と誤判定）が露呈した。一部は修正済みだが、つぎはぎが続く。

ゴール：**「先に来るべき」という制約を全部"辺"にして1つのグラフを作り、1回のトポロジカルソートで最適順を出す**——LOOT が Skyrim 250MOD を手動ルールゼロで安定させているのと同じ原理を、Reloaded 向けに最適化する。**新ゲーム・新MODにも永続的に効く一般ルール**だけで構成し、個別MODの手チューニングはしない。

参考資料：`C:\Users\rainb\md\mod`（LOOT のソートアルゴリズム / Vortex のアプローチ等）。要点は [[reference-loot-vortex-autosort]] に記録。

## 北極星（揺らさない原則）

- **完全自動。** ユーザーが触るのは導入・有効/無効・設定のみ。並び替えはしない・できない（手動UIなし）。順序をユーザーに聞かない（"要確認"なし）。
- **ソフト自身が解決する"仕組み"。** 修正は必ず**どのゲーム・どのMODにも効く一般ルール**。MOD名のハードコード・特定MOD向け分岐は禁止。Claude が個別MODを手で直すのではなく、ソフトのルールが解く。
- **唯一ユーザーに見せる競合 = 冗長MOD**（資源が全部他に上書きされて無意味になっているMOD）→ 無効化で解決。Vortex の "redundant mod" 通知に相当。
- **表示ソート（カテゴリ/作者/名前）は閲覧用で実配置と独立。**
- 不可侵ルール継承：実行時に外部AIを呼ばない（無料・オフライン）／失敗を握りつぶさない／自動修正前にバックアップ＋ワンタップ復元／ユーザーデータは `%APPDATA%\ReloadedHelper` のみ。

## 確定事実（実機・調査で裏取り済み）

- **Reloaded は「リストの下＝後に読み込み＝ファイル競合で勝つ」**（last-wins）。順序は実際に効く（"反映されない"はユーザーの勘違いと判明）。出典：LOOT/P4G CEP/P3R ガイド。
- Reloaded は **MO2型（リスト順＝競合の勝敗）**であり、Vortex型（順序無関係・手動ルールのみ）ではない。よって「正しい順序を自動で作る」ことが解。
- **LOOT は説明文を読まない。** 当ソフトも 2026-06-25 にキーワード照合を名前のみへ縮小（同方向）。

## アーキテクチャ：統一制約グラフ

頂点＝有効MOD。辺＝「X は Y より前に読む」という制約。**全ての辺を1つのグラフに集め、循環を除去し、安定トポロジカルソートで1本の順序を出す。**

### 辺の4つの源（強い順。弱い辺は循環を作るなら追加しない＝強い制約が勝つ）

1. **依存の辺（ハード）** … MOD が依存先に B を挙げていれば `B → 自分`（B が前）。`ModConfig` の `ModDependencies`／`PluginData` 依存マップから。**絶対に守る。**
2. **グループの辺（大枠の骨格）** … 各MODを順序付きグループ（土台 → ゲーム機能 → 戦闘 → … → 見た目上書き）に分類し、「早いグループの全MOD → 遅いグループの全MOD」へ辺。これが「土台前・見た目後」の骨格。グループ＝種類（`ModType`）。
3. **重なりの辺（自動裁定の核）** … 2つのMODが同じ資源（同一の `file:`/`song:`/`costume:` キー）を触るとき、**「触る資源が多いMOD → 少ないMOD」**へ辺（多い方が前、**少ない方が後＝勝ち**）。LOOT 実証済みの一般則：狙い撃ちの小さいMODが、広く触る大きいMODに勝つ。MODの中身を知らずに競合を自動解決。
4. **タイブレーク（安定性）** … 上記で順序が定まらない部分は**現在順を保つ**。これは安定トポロジカルソート（現在順インデックスを優先度に使う Kahn 法）で自然に実現＝**無関係MODは動かさない**。

> **グループ骨格についての設計判断：** フェーズ1は「競合だけ動かす（churnゼロ）」を強く志向したが、それは"分類が間違っていた"反動だった。分類が正しくなった今、LOOT 同様に**正しいグループ骨格を課す**方が信頼でき・世界標準とも一致する。タイブレークが「制約が許す範囲では現在順を保つ」ため、動くのは"正しい場所へ寄せる"ぶんだけで、無意味な全体シャッフルにはならない。

### 循環の扱い

依存の辺は循環しない前提（すれば導入エラーとして既存どおり末尾に原順で残す）。グループ・重なりの辺は、**追加すると循環する場合はその辺を捨てる**（強い辺が既にある＝強い制約が勝つ）。最終グラフが非循環であることを確認してからソート。

### 出力

並び替え後の順序＋**「何を・なぜ動かしたか／各MODのグループ判定理由」**のリスト（診断パネルの履歴へ）。

## グループ（種類）の決め方

「土台か・見た目か」をMODの中身から一般ルールで判定する（優先順）：

1. **`IsLibrary` 指定** → 土台（Library）。
2. **依存グラフによる枠組み検出（2026-06-25 実装済み）** … **多数（≥2）のMODから依存される＋自分はゲームファイル（資源）を持たない → 土台。** 例：modloader(依存144)・crifs.v2.hook(25)・CostumeFramework(29)・BGME.Framework・Ryo 等。資源を大量に持つ人気コンテンツ（パッチ多数のスキン）とは「資源ゼロ」で区別。**これが汎用の肝**——どのゲーム・どのMODでも効く。
3. **実際に触る資源** … `song:` を書く→音楽、`costume:` を登録→衣装（確実な手がかり）。
4. **GameBananaカテゴリ**（あれば）… Battles→戦闘、Portraits→立ち絵、User Interface→UI、Texture Packs→スキン 等（実機で良好）。
5. **名前の語**（説明文は読まない）… 最後の保険。
6. 手がかり無し → 末尾グループ（Unknown）。ただし依存されていれば上の2で土台になる。

グループの順序＝`ModTypeInfo.Rank`（土台0 … 見た目側が後）。これがグループ辺の向きを決める。

## 重なりの辺＝自動裁定（詳細）

- 2MODが**1つでも同じ資源キーを共有**＝重なり。既存 `ConflictDetector`／資源インデックスで対を抽出。
- 各重なり対で、**触る資源の総数が多い方 → 少ない方**へ辺（少ない方が後＝勝ち）。総数が同じ／重ならないなら辺なし（タイブレークに委ねる）。
- これにより「双葉の顔1枚だけ差し替えるMOD」が「双葉まるごと大型MOD」に勝つ、が**自動で・MODを知らずに**決まる。
- 依存・グループの辺と矛盾する重なり辺は循環回避で捨てられる（強い制約が優先）。

---

## 冗長MOD検出（唯一のユーザー向け通知）

- 最終順序の下で、**あるMODの資源が全て他MODに上書きされ、可視寄与がゼロ**＝冗長（Vortex の "redundant" と同じ）。または資源を1つも持たない有効MOD。
- **非ブロッキングに提示**：「このMODは他に全部上書きされ、今は無意味です。無効化しますか？」解決は**無効化のみ**（並び替えはさせない）。却下も記憶し二度警告しない。
- これが「片方だけ有効想定」の変種ペアも自然に拾う（一方が他方を完全に覆う）。

## 新規・変更・再利用クラス

**再利用（フェーズ1の資産）**
- `Analyzers/*`（資源抽出）・`ConflictDetector`（資源キー重なり）・依存パース・`ModTypeClassifier`（グループ＝種類、枠組み検出込み）・`AutoSortCoordinator`（バックアップ＋適用＋履歴）・`LoadOrderBackupService`・`GameDiagnostics`・`PreferenceStore`（冗長警告の却下記憶へ転用）。
- 実機ダンプ診断（`_RealInstallDump` の発展）＝**常設の検証道具**に格上げ。

**新規**
- `ConstraintGraph`（新規・中核）… 4種の辺を集め、循環回避で追加し、安定トポロジカルソート。`LoadOrderSorter`（既存の安定 Kahn 法）を一般辺対応に拡張 or 内包。
- `GroupModel`（新規・小）… `ModType` → グループ順・グループ辺の生成。
- `OverlapEdges`（新規・小）… 資源重なり対 →「多い→少ない」辺。
- `RedundancyDetector`（新規）… 完全被上書き／資源ゼロの検出。

**変更/置換**
- `LoadOrderOptimizer` … ad-hoc な「層整列＋競合移動」を**統一グラフ呼び出し**へ置換。出力（順序＋配置理由）の形は維持。
- `MainViewModel.RunAutoSort` … 新エンジンへ配線（資源・依存・グループ・重なりを渡す）。

## 一般性の担保（"資産"であること）

- **MOD名・特定タイトルのハードコード禁止。** 全ての辺ルールはゲーム/MOD非依存（依存数・資源数・資源キー一致・グループ順）。
- 新ゲーム（P4G/P3R/未来作）でも redirect ルート等が違うだけで**同じエンジンが効く**。ゲーム固有は「資源の読み方（アナライザ）」に閉じ込める。
- **検証は実機ダンプで**：実 `C:\FreeSoft\Reloaded-II` の全MODに対し、グループ判定と最終順序を before→after でダンプ確認してから「良くなった」と言う（合成テスト緑だけで判断しない＝今回の教訓）。

## 勝者方向（last-wins）

「読み込み順で後＝勝ち」を採用（確定事実）。資源の勝敗判定は1箇所に集約し、万一逆の枠組みがあれば反転で対応。

## エラーハンドリング

| 状況 | 対処 |
|---|---|
| アナライザが形式を読めない | 「未対応」として記録（事実を捏造しない・既存 `ModAnalysis.Failures`） |
| 循環（依存以外の辺由来） | 弱い辺を落として回避。落とした事実を記録 |
| 依存の循環 | 末尾に原順で残す（既存） |
| 適用失敗 | バックアップから復帰＋履歴に理由 |

## テスト方針

- Core 各部（`ConstraintGraph`・`GroupModel`・`OverlapEdges`・`RedundancyDetector`・拡張ソート）は xUnit。**辺の優先順位（依存＞グループ＞重なり＞安定）と循環回避をケースで固定。**
- 重なり裁定「少数派が勝つ」、グループ骨格、依存厳守を個別ケースで。
- **実機ダンプ検証を必須ゲートに**（合成テスト緑＝完了、ではない）。実159MODでグループ・順序・冗長検出を目視。
- UI（冗長提示・配置理由）は実機目視。

## スコープ外（将来反復）
