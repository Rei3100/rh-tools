# 完成度向上フェーズ（信頼性オーバーホール）設計書

## このフェーズの目的

ソフトを「信頼して任せられる」状態にする。今は機能は載っているが、取得が実質動かず・失敗が見えず・UIが不統一で、ユーザーが不安を抱える。
最終ゴールは **「MODを入れて有効にして遊ぶ、それだけ」**。配置順・競合・反映されない理由・落ちる理由——その面倒を**ソフト自身が**考えて解決する。

過去はフェーズを細かく分けてきたが、今回は A〜E を1つの大型まとまりとして進める（ユーザー合意）。

## 全体方針（横断ルール）

1. **実行時に Claude を一切呼ばない。** 賢さはソフトの中（ルール＋MOD内容の解析）に持つ。Claude 依存にしない。資産として育つ拡張型にする。
2. **失敗を握りつぶさない。** 現状の `catch { }` 多用をやめ、成功/失敗とその理由を必ずログ＋画面に残す（"見える化"）。
3. **自動修正は安全なものだけ、迷うものは警告。** 自動修正の前に必ずバックアップし、ワンタップで元に戻せる。
4. **処理は逐次反映。** 一括処理の完了を待たず、1件終わるごとに即UIへ反映する。
5. **本体側では何もしない・全部ヘルパーに集約。** 設定・診断・修正はすべてヘルパーから。

## 対象ゲーム

ペルソナ5 ザ・ロイヤル / ペルソナ4 ザ・ゴールデン / ペルソナ5スクランブル。
（インストール先実測: `C:\FreeSoft\Reloaded-II`、MOD 310 件）

# セクションA：データ取得・翻訳の作り直し（精度の本丸・最優先）

## 確定した根本原因（実証済み）

userdata.json 実測で GameBananaId・Category がほぼ全件 null。原因を実機で特定：

1. **API の URL とホストが誤り。** 現状 `api.gamebanana.com/apiv11/Mod/{id}/ProfilePage?fields=...` を呼ぶが、apiv11 の正しいホストは `gamebanana.com`（`api.` を付けると 404）。
2. **応答形式の解釈が誤り。** コードは平配列 `["名前","説明","カテゴリ","ゲームID"]` を期待するが、apiv11 はオブジェクト（`_sName`・`_aCategory._sName`…）を返す。→ パース失敗 → null。
3. **ニワトリ卵問題。** MOD名検索は「GameBanana ID 既知の MOD が1個以上ある」ことが前提。URL 欠落 MOD ばかりだと検索が永遠に走らない。
4. **検索キーが化けている。** 過去の取得で ModConfig.json の ModName が日本語に上書き済み。その日本語で GameBanana を検索 → 失敗。
5. **失敗の全握りつぶし。** 何が成功/失敗したかユーザーにもログにも残らない。

## 直し方

### A-1. API 修正（apiv11 `_csvProperties`）
- エンドポイントを `https://gamebanana.com/apiv11/Mod/{id}?_csvProperties=_sName,_aCategory,_aGame,_aSubmitter,_sText` 形式に修正。
- 取得項目：**名前 `_sName`／説明 `_sText`／カテゴリ `_aCategory._sName`／ゲームID `_aGame._idRow`／作者 `_aSubmitter._sName`**。
- 検索は apiv11 `Util/Search/Results` を継続利用。応答形式は実機で確認しつつ実装（TDD で実応答に対して検証）。

### A-2. ゲームID内蔵でニワトリ卵を解消
- P5R / P4G / P5S の GameBanana ゲームIDをソフトに内蔵（起動時に外部依存しない）。
- 内蔵IDで最初からゲーム横断検索が可能 → **URL 欠落 MOD も名前検索で拾える**。
- 将来ゲームが増えた場合に追記しやすい形（定数テーブル）にする。

### A-3. マッチングの安定化（化けた名前で検索しない）
- 検索キーは **ModId と「元の英語名」**。元の英語名（ModConfig.json 上書き前の値）を userdata.json にスナップショット保存し、再取得しても安定させる。
- マッチ手順：① ProjectUrl から直接ID抽出 → ② ModId/英語名で内蔵ゲームID検索（類似度しきい値） → ③ マッチ無し＝翻訳のみ。

### A-4. 全件強制取り直し ＋ 以後は差分
- 「全件強制取り直し」操作：既存翻訳の有無に関係なく GameBanana から取り直す。
  - GB マッチ → GB の名前・説明を翻訳＋用語辞書補正、カテゴリ・作者を取得。
  - GB マッチ無し → 英語原本を機械翻訳。
  - 翻訳も失敗 → 英語のまま（**古い日本語を黙って残さない**）。
- 通常は差分のみ（新規・バージョン変化した MOD だけ）。毎回 310 件を翻訳して無駄打ちしない。

### A-5. 見える化（更新レポート）
- MOD ごとに結果を記録：`GB一致(ID) / 翻訳のみ / 失敗(理由)`。
- ログファイル（`%APPDATA%\ReloadedHelper\update-log.txt` 等）へ追記＋画面に「更新レポート」パネルで提示。
- これが「信頼できる」の中核。何が起きたかが常に分かる。

### A-6. 逐次反映（重要）
- 一括処理中、**1件処理し終わるごとにその MOD だけ即リストへ反映**（行を差し替え）。完了を待たない。
- 画面を丸ごと作り直さない（`LoadFrom` の全再構築をやめ、対象行のみ更新）。これにより B のタブ切替バグも同時に解消。
- 進捗表示「更新中 N/総数 件」を継続表示。

### A-7. カテゴリ・作者
- GameBanana カテゴリを日本語バッジ表示（既存マッピング活用、未知値はそのまま表示）。
- 作者は GameBanana 投稿者名で補完。→ 作者分け・カテゴリ分けが機能する。

## A の新規/変更クラス（想定）
- `GameBananaClient` … エンドポイント・パース全面修正、作者・ゲームID対応。
- `GameRegistry`（新規）… 内蔵ゲームIDテーブル。
- `ModUserData` … `OriginalName`（英語原本）追加。
- `MetadataRefreshService`（新規 or App 内ロジック整理）… 逐次反映・レポート生成。
- App 側：全件強制取り直しボタン、更新レポートパネル、行単位更新。

# セクションB：UI統一＋バグ修正

## B-1. タイトルバー
- 最上部（ソフト名・最小化等）が標準ライトのまま。**カスタムタイトルバー**化し、本体UIの配色と完全一致させる。

## B-2. メニュー・ポップアップの統一
- 右クリックメニュー／システムトレイメニュー／更新ポップアップを全てダーク配色・読みやすいサイズに統一。
- 更新ポップアップは素の Windows ダイアログをやめ、**本体UIと同じ見た目の自前ダイアログ**へ差し替え（"浮いてる・芋っぽい"の解消）。
- システムトレイメニューは過去対応後も違和感が残るため、表示・配色を作り直す。

## B-3. タブ勝手切替バグ
- 再描画前に「現在選択中のゲーム(AppId)」を保持し、描画後に必ず復元。
- 併せて A-6 の行単位更新で全再構築自体を減らし、再発を防ぐ。

# セクションC：MOD個別設定（ヘルパー側から日本語で）

## 実態
- 各 MOD の設定値は `User\Mods\<MOD名>\Config.json` に素の JSON で存在（例 `"DisableVictoryBgm": false`、`"BaseBgmId_P5R": 12000`）。読み書き可能。

## C-1. 自動フォーム化
- Config.json を読み、値の型から入力UIを自動生成：真偽→トグル、数値→数値欄、文字列→入力欄（選択肢が判明するものはプルダウン）。
- 保存時は JSON の型・他フィールドを保って書き戻し。

## C-2. 開き方
- MOD編集画面／右クリックメニューから「設定」を開く。設定ファイルが無い MOD は項目を出さない。

## C-3. 正直な限界
- 項目名の日本語は機械翻訳のベストエフォート（意味・説明は本体DLL内のため完全には取れない）。読み書き自体は確実に動く。camelCase を分割して翻訳＋用語辞書補正で可読性を上げる。

# セクションD+E：診断・自動解決エンジン（最も野心的・拡張型）

D（並び替え精度）と E（クラッシュ・解析）は同一目的＝**「面倒を考えなくていい」**。MOD のファイル中身を読んで「このMODがゲームの何を触るか」を割り出し、競合・反映されない理由・落ちる理由を**ソフト自力で**診断・解決する。

## 線引き（重要）
- ここは終わりのない領域（MOD種は無限に増える）。今回は **「土台＋よくある高価値の診断」** を作り、以後は**アナライザを足すだけで賢くなる拡張型**にする。AI を呼ばず、ソフトに知識（ルール）が積み上がる構造。

## エンジン構成

### E-1. MOD内容スキャナ
- 各 MOD のファイルツリーを読み、「差し替える／追加するゲーム内ファイル」と、フレームワーク型 MOD の登録内容（曲ID・コスチューム等）を抽出。
- 仕組み別アナライザ（拡張ポイント）：
  - **ファイル置換型**（多数）：対象ファイルパス一覧を抽出。
  - **BGME 音楽型**：変更する曲IDを抽出。
  - **コスチュームフレームワーク型**：登録コスチュームを抽出。

### E-2. 競合・被り検出
- 同じ対象ファイル／同じ曲ID を触る複数 MOD を検出し、「被り」と読み込み順の影響を提示。

### E-3.「なぜ効かない」診断
- 典型原因を自動判定して日本語で提示：依存フレームワークが無効／対象ゲーム違い（AppId 不一致）／読み込み順で別MODに上書き／同じファイルを上書きする MOD の存在。
- 例（解決対象の"種類"）：髪型MODが効かない＝同ファイルを上書きする MOD を特定。

### E-4. 読み込み順の自動最適化
- 依存トポロジー＋競合情報＋仕組み別ルール（例：後勝ち/先勝ち）で精度を上げる。
- **安全に正解が定まるものは自動修正、人の好みが絡むもの（どちらを優先か）は警告して選択させる**。
- 自動修正前にバックアップ（既存のロードオーダーバックアップ機構を活用）、ワンタップで元に戻せる。

### E-5. クラッシュ検知
- Reloaded はクラッシュログをディスク保存しないため、**Windows アプリケーションイベントログ**からゲーム exe の異常終了を検知。
- 将来：ヘルパー経由でゲーム起動時に Reloaded のコンソール出力を保存し、原因解析の材料にする（拡張）。

## D+E の新規クラス（想定）
- `ModContentScanner`（新規）… ファイルツリー/登録内容の抽出。
- `IModAnalyzer` 群（新規・拡張ポイント）… 仕組み別解析。
- `ConflictDetector`（新規）… 被り・競合検出。
- `LoadOrderSorter` 拡張 … 競合情報を反映した最適化。
- `CrashLogReader`（新規）… Windows イベントログ参照。
- `DiagnosticsService`（新規）… 診断結果の集約・自動修正/警告の振り分け。

# 実装順（このまとまり内の優先度）

1. **A（取得精度）** … 最優先。最大の不満であり、カテゴリ・作者・依存情報は D+E の入力にもなる土台。
2. **B（UI・バグ）** … A の逐次反映と連動するタブ切替修正を含む。見た目で信頼回復。
3. **C（個別設定）** … 独立性が高い新機能。
4. **D+E（診断エンジン）** … 土台＋高価値診断を構築し、拡張型として育てる。

# エラーハンドリング方針

| 状況 | 対処 |
|------|------|
| GameBanana API 失敗 | タイムアウト後スキップ。レポートに「失敗(理由)」記録（握りつぶさない） |
| 翻訳 API 失敗 | 1回リトライ→失敗なら英語のまま。レポート記録 |
| ModConfig/Config 書き込み失敗 | レポート記録しスキップ（起動・処理は継続） |
| 自動修正の対象が曖昧 | 自動修正せず警告。ユーザー選択 |
| 自動修正実施 | 事前バックアップ＋元に戻す導線を提供 |

# テスト方針

- Core 層（`GameBananaClient`・`ModContentScanner`・`ConflictDetector`・`CrashLogReader`・Config 読み書き等）は xUnit でユニットテスト。HTTP は DI でモック化。
- GameBanana/翻訳の実応答に対する解析は、実取得サンプルを固定データ化して検証。
- 翻訳精度・UI 見た目は目視確認（自動テスト対象外）。
